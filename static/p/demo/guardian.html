The Guardian Lab Relay is a thin HTTP/SSE relay that runs Claude Code sessions from the Guardian app against the live Guardian project environment on the B650 host. It pins cwd to the active Guardian project so MCP and config are present, persists conversations and turns to SQLite, and shapes raw agent behavior into product-friendly streaming events for the app.

It is intentionally small. It is not meant to become the long-term canonical cloud backend for the product — that role is reserved for GCD.

Mission

The relay should be the best place to:

  • Run Claude/agent sessions against the live Guardian environment.
  • Prototype app-to-agent interaction models quickly.
  • Experiment with warm-session and memory-loading strategies.
  • Shape streaming behavior into product-friendly UX contracts.
  • Orchestrate Guardian tools and MCP integrations.
  • Keep relay continuity state durable and testable.
  • Support trusted-user demos, research, and fast iteration.

The lab relay is a stepping stone, not a dead end. We want to learn fast locally without creating migration debt everywhere.

Architecture split

Three responsibilities, three owners:

Relay canonical for…

  • Claude session continuity
  • Durable synced conversation history
  • Turn execution and provenance

App canonical for…

  • Rendering
  • Local cache and offline projection
  • Optimistic UI
  • Branch UX and higher-level interaction behavior

GCD future canonical for…

  • Auth (token issuance, validation)
  • Policy
  • Cloud coordination across multiple devices and users

Until GCD exists, the current header-based x-guardian-user-id model is acceptable for trusted beta use, but it should be treated as temporary.

SSE streaming contract

The relay streams JSON payloads as SSE data: lines and terminates with data: [DONE]. Tool activity events include normalized phase metadata so the app can surface progress for long-running memory queries:

{
  "type": "tool_start",
  "tool": "mcp__guardian-memory__search_memory",
  "toolName": "mcp__guardian-memory__search_memory",
  "phase": "searching",
  "label": "searching memory"
}

Current phases: discovering · searching · reading · working. Primary event types are init, queued, token, tool_start, turn_promoted, result, and error. The full contract lives in docs/sse-contract.md.

Warm sessions

The relay supports hidden per-user warm sessions so new conversations can resume a preloaded Guardian-aware Claude context without any app API changes.

Current policy:

  • One hidden warm base session per user.
  • Refresh every 50m by default.
  • Fork new conversations from that base.
  • Target a rich working set inside the standard context budget.
  • Live model: claude-opus-4-6[1m] · warmup model: claude-opus-4-6[1m] · fast mode: enabled.

Why this matters

Warm sessions are one of the highest-leverage experiments the relay enables. They are highly product-specific, tightly coupled to Guardian memory behavior, and easiest to explore on the machine that actually runs the agent and tools.

Deployment topology

The B650 relay runs under user systemd services so it comes back after power loss or reboot. Current live and target topologies:

Slot Service unit Port Role
Router guardian-lab-router.service 3010 Frontend, slot switching
Stable blue guardian-lab-relay@stable-blue.service 3000 Primary stable lane
Stable green guardian-lab-relay@stable-green.service 3002 Cold standby on shared SQLite
Experimental guardian-lab-relay@experimental.service 3001 Active protocol experiments

Operational note. stable-blue and stable-green share a single stable SQLite data plane. With the current SQLite relay, only one stable slot should run at a time — slot switches are cold-standby cutovers over shared stable state. The router stays in front.

Service management

Typical commands on the B650:

systemctl --user daemon-reload
systemctl --user enable \
  guardian-lab-relay@stable-blue.service \
  guardian-lab-relay@stable-green.service \
  guardian-lab-relay@experimental.service \
  guardian-lab-router.service

systemctl --user start  guardian-lab-relay@stable-blue.service
systemctl --user status guardian-lab-relay@stable-blue.service --no-pager
journalctl --user -u guardian-lab-relay@stable-blue.service -n 100 --no-pager

Verification

End-to-end harness against the live B650 relay:

RELAY_URL=http://100.66.64.118:3000 \
  ENABLE_GUARDIAN_MEMORY_CHECK=1 \
  npm test

The harness includes an opt-in Guardian-memory identity milestone that verifies the relay can return Jackie-specific facts through the real Guardian MCP path. Current known-good live results:

  • Relay harness: 55 passed, 0 failed
  • Restart continuity: proven on B650
  • Auto-title generation: proven on B650
  • Tailnet HTTPS: live

What the relay is NOT

We should be much more careful about expanding the relay into:

  • The canonical product auth system
  • The canonical multi-device sync backend
  • The canonical transcript database (beyond execution provenance)
  • The full cloud control plane

Current intentional limits:

  • Pending prompts are not persisted across restart.
  • Transient streaming/placeholder UI state is not yet the canonical relay transcript.
  • App still owns optimistic UX, local cache/offline projection, and branch behavior.
