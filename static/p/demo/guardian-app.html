The Guardian app is the front door to Guardian — a persistent shell with three first-class work surfaces (New Chat, Chat, Field Notes), a launcher drawer, and a closed-loop dev harness. It talks to the Guardian Lab Relay over HTTP/SSE and consumes the relay's durable-turns lifecycle.

The app owns rendering, optimistic UI, local cache, and branch UX. It does not own Claude session continuity, conversation history, or turn execution — those are relay-canonical.

Overview

Haptica is not best modelled as "a screen with a menu overlay." It is better modelled as three layers:

  • A navigation layer
  • A foreground workspace layer
  • A small utility layer

The drawer is a launcher into first-class work surfaces. It is not the product itself. Swiping reveals the navigation layer by pulling the workspace aside — closer to the feel of premium mobile apps than a simple left panel sliding on top of content.

We own product structure, branding, and content presentation. We do not want to own low-level drawer gesture arbitration long-term.

Shell architecture

Four ownership layers

Each layer has a clear owner and responsibilities. Cross-layer ownership is the source of every recurring shell bug we've shipped.

LayerOwnerResponsibilities
Router expo-router Route tree, deep-link mapping, screen identity, back-stack
Shell expo-router/drawer Drawer state, reveal behavior, gesture arbitration
Screen Route screen files Content rendering, screen-local actions and state
Presentation Haptica theme Visual identity, drawer content, typography, badges

Canonical route structure

app/_layout.tsx                  # global providers only
app/(drawer)/_layout.tsx         # official drawer shell
app/(drawer)/index.tsx           # New Chat / landing workspace
app/(drawer)/chat/[id].tsx       # Chat workspace
app/(drawer)/dashboard.tsx       # Field Notes workspace
app/(drawer)/settings.tsx        # Settings workspace
app/debug.tsx                    # internal tooling, outside drawer

Primary work surfaces

These are first-class destinations and should feel like peers:

  • New Chat — landing workspace, optimistic first-message handoff
  • Chat — active conversation, durable-turns lifecycle, queued/running indicators
  • Field Notes — dashboard surface, peer of Chat (not a secondary admin surface)

Settings and Debug are utility surfaces. They should not distort the primary shell model.

Drawer model

The shell uses a reveal / pull-away presentation model:

  • The foreground workspace moves.
  • The navigation layer is revealed behind it.
  • The shell feels physically connected to the workspace.

What we avoid: cheap overlay feel, drawer as a detached panel on top of content, content feeling inert while the menu simply covers it.

Gesture rules

These are normative rules, not tuning suggestions:

  1. Vertical scroll wins in scrollable content. Drawer gestures must not sit on top of full scroll regions.
  2. Drawer open is shell-owned. No custom whole-app pan wrappers as the long-term mechanism.
  3. Drawer close is shell-owned. No layered full-panel custom close gestures over scrollable drawer content.
  4. Tuning happens at the official drawer layer — swipeEnabled, swipeEdgeWidth, swipeMinDistance, configureGestureHandler.
  5. The drawer gesture is infrastructure, not a product innovation surface. Invisible, reliable, conventional.

Back behavior

  1. If the drawer is open, the first back action closes the drawer.
  2. If the drawer is closed, back operates on content/navigation history.
  3. Switching between top-level work surfaces must not create ghost back-stack states.

Closed-loop dev harness

./scripts/device.sh is the autonomous device-interaction harness for screenshots, UI driving, and state verification. The core principle: do not trust a system because it says it is healthy. Trust it only after you inject a controlled stimulus and recover that exact stimulus through the system's real output surfaces. Artifacts beat assumptions.

Daily workflow — three commands

./scripts/device.sh screenshot [device]   # → "file.png (b4 c209d5f2 · 3min old · live)"
./scripts/device.sh web [--pixel|--s24|--iphone15pro]
./scripts/device.sh uifind [device] <text>   # find element + tap coords

Every screenshot is self-documenting. The in-app badge shows b4·c209d5f2 · live (build number + source hash + backend). The harness output prints staleness inline. Always use uifind to locate elements before tapping — never guess coordinates.

Why no hot reload

Always cold-restart the app and browser after code changes. Hot reload preserves JavaScript module state across reloads, causing silent bugs that don't exist in a clean launch:

  • Singleton state stacks. Module-level variables aren't reset; each reload creates new instances alongside the old ones — duplicate event emissions, stale backend references, leaked timers.
  • Intervals and timers accumulate. The auto-backend probe runs every 30s via setInterval; after 5 reloads you have 5 concurrent probe loops.
  • Database connections leak. SQLite connections from the previous bundle stay open. Eventually you hit FD limits or lock contention.
  • Browser state drifts. WebSocket connections to Metro go stale silently; service worker cache can serve old bundles; IndexedDB/localStorage persists with potentially incompatible schemas.

Emulatorkill → launchForce stop + cold start

Cold restart adds ~5s vs hot reload's ~1s. That's worth it — a hot-reload bug that takes 20 min to diagnose costs far more than the 4s you saved.

Working with the lab relay

The app and the Guardian Lab Relay coordinate over a small set of stable contracts. The app sends additive request headers identifying its lane, slot preference, version, build, platform, and capabilities. The relay reports its assigned identity through /readyz and /health.

Headers the app sends

x-guardian-user-id          # tenant scope (temporary, until GCD)
x-guardian-app-version
x-guardian-build
x-guardian-platform         # ios | android | web
x-guardian-client-lane      # stable | experimental
x-guardian-client-slot-preference
x-guardian-capabilities     # durable_turns_v1, queued_turns_v1, ...
x-request-id                # per-request correlation
x-idempotency-key           # on POST /api/conversations and /messages

Fields the relay reports back

relayVersion · lane · slot · model · warmModel
capabilities · featureFlags · activeTurnCount

Settings shows requested vs assigned routing separately, so rollout debugging stays honest. The app can observe assigned routing but cannot make it authoritative on its own — the router/relay side is the source of truth for assignment.

Stream failure classification

The app maps relay error.code values into clearer UX:

  • relay_draining → degraded ("guardian is updating")
  • upstream_network → degraded, upstream-connection wording
  • upstream_auth → degraded, upstream-auth wording
  • conversation_not_found / no_pending_prompt → resend/sync wording without flagging the whole app as offline

Read paths fall back to a local cached mirror when relay fetches fail, with explicit relay / degraded / cached / offline states surfaced in composer copy. See the relay's SSE contract for the full event vocabulary.
